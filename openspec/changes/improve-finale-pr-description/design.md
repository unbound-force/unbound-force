## Context

The `/finale` command (`.opencode/commands/finale.md`)
wraps end-of-branch workflow: stage, commit, push, create
PR, watch CI, return to main. Step 3 generates commit
messages, Step 5 generates PR title and body.

Current Step 5c produces a minimal `## Summary` with
bullet points from `git log`. Step 3 generates a
conventional commit message with no attribution. PR
creation uses inline `--body` shell interpolation.

The proposal (proposal.md) identifies six improvements
addressing Constitution Principles I, III, and V.

## Goals / Non-Goals

### Goals
- Produce structured PR descriptions with sections that
  reviewers can navigate predictably
- Add AI attribution to both PR descriptions and commit
  messages for provenance traceability (Principle III)
- Surface unresolved review-council findings so reviewers
  have context on known issues
- Detect and respect repository PR templates for cross-
  repo consistency
- Eliminate the shell injection vector in PR creation
  (Principle V)

### Non-Goals
- AI-recorded demo attachment (split to #287)
- Changes to `/unleash` or `/review-pr` commands
- Creating a PR template for this repository (separate
  concern)
- Changing the commit message approval UX (the existing
  "Approve, edit, or provide your own?" flow is preserved)
- PR title generation improvements (issue #142 mentions
  "name" consistency — the PR title generation in Step 5b
  is adequate and out of scope for this change)

## Decisions

### D1: Structured PR Body Sections

**Decision**: Add four sections to the generated PR body:
`## Summary` (what was done), `## How to Test`
(verification steps), `## How to Demo` (demo walkthrough),
`## Key Files Changed` (file listing with brief
descriptions).

**Rationale**: Mirrors the proven pattern from `/unleash`
Step 8 (Demo). Reviewers get consistent, navigable PR
descriptions. Sections are populated from commit history,
spec artifacts (if on an OpenSpec/Speckit branch), and
diff analysis.

**Constitution alignment**: Principle I — self-describing
artifacts that consumers can interpret without consulting
the producer.

### D2: PR Template Detection

**Decision**: Before generating the default structured
body, check for `.github/PULL_REQUEST_TEMPLATE.md` (case-
insensitive: also `.github/pull_request_template.md`). If
found, read the template and use its section structure as
the skeleton, mapping generated content to matching
sections. If no template exists, fall back to the default
structured format from D1.

**Rationale**: Respecting PR templates (per
marcusburghardt's suggestion on #142) makes `/finale`
composable across repos with different conventions
(Principle II). The template becomes a configuration
surface.

**Trade-off**: Template parsing adds complexity. Mitigated
by a simple heuristic: match `##` headings in the template
and populate content under matching sections. Unmatched
template sections are preserved as-is for the user to fill
in.

### D3: AI Attribution in PR Descriptions

**Decision**: Append an attribution footer to every
`/finale`-generated PR body:

```
_This PR was generated by /finale (AI-assisted)._
```

**Rationale**: Consistent with the pattern in `/review-pr`
(line 967-968). Satisfies Principle III provenance
requirement. Inserted before the user confirmation step so
the user can modify or remove it.

### D4: AI Attribution in Commit Messages

**Decision**: Add two forms of attribution to `/finale`-
generated commit messages:

1. **Git trailer** (machine-parseable):
   ```
   AI-assisted-by: /finale
   ```
2. **Footer line** (human-readable):
   ```
   Generated with AI assistance (/finale)
   ```

Both are appended after the commit body, separated by a
blank line. The trailer follows git's standard
`Key: Value` format and is parseable by `git log
--format='%(trailers)'`.

**Rationale**: Dual format serves both tooling (trailer)
and human readers (footer). The trailer enables automated
analysis of AI-generated vs human-written commits.
Attribution is included in the proposed commit message
shown to the user for approval — the user can edit or
remove it.

### D5: Review-Council Findings in PR Body

**Decision**: When generating the PR body, check for a
review-council output marker. If unresolved findings
exist, add a `## Known Issues` section listing them with
severity and brief description.

**Source of findings**: The `/review-council` command
currently outputs findings to the terminal only. This
design requires `/finale` to look for findings in the
conversation context (the agent's memory of prior
`/review-council` output in the same session). If no
review-council was run, the section is omitted.

**Rationale**: Principle I — artifact-based communication
between commands. Reviewers get visibility into issues the
AI could not resolve, reducing duplicate review effort.

**Trade-off**: Depends on session-level context rather than
a persisted artifact. A future enhancement could persist
review-council output to a file, but that is out of scope
for this change.

### D6: Shell Injection Fix

**Decision**: Replace inline `--body` interpolation:

```bash
# Before (unsafe):
gh pr create --title "<title>" --body "<body>"

# After (safe):
gh pr create --title "<title>" --body-file "<temp-file>"
```

Write the PR body to a temporary file and use `--body-file`
for the body. The title is kept as `--title` since it is a
single line validated through user confirmation.

**Rationale**: Principle V — input validation before shell
execution. AI-generated PR body content can contain shell
metacharacters (`"`, `$`, `` ` ``). The `--body-file`
pattern avoids shell interpolation entirely, consistent
with the established `mktemp` + cleanup pattern used in
`/triage-issue` and `/address-feedback`.

**Implementation**: Write to `mktemp`, `chmod 600`, use
`--body-file`, then `rm -f` in all exit paths.

## Risks / Trade-offs

1. **PR template parsing fragility**: Simple `##` heading
   matching may miss templates with non-standard structure.
   Mitigated by falling back to default format when
   template parsing fails.

2. **Review-council findings availability**: Depends on
   session context (prior `/review-council` run in the
   same conversation). If no findings are available, the
   section is silently omitted. A persisted artifact
   mechanism is a future enhancement.

3. **Cross-repo scaffold sync**: Changes to `finale.md`
   propagate to all hero repos via `uf init`. The new
   structured format becomes the default across all repos.
   This is intentional (consistency) but means all repos
   adopt the new PR format simultaneously.

4. **Attribution opt-out**: Users can remove attribution
   during the approval step. This is by design — the
   user confirmation gate is preserved. There is no
   enforcement mechanism to prevent attribution removal.
