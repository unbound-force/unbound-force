## Context

`/review-council` is a local, pre-PR multi-agent review command
that delegates to all discovered Divisor persona agents, collects
findings, runs a fix loop (up to 3 iterations), and produces a
terminal-only final report. It currently has no GitHub
integration.

`/review-pr` is a single-agent, post-PR review command that
includes GitHub review posting (Step 11) with review state
awareness (Step 7.5). Its posting mechanism is proven and handles
edge cases (duplicate detection, stale reviews, CODEOWNER checks,
graceful degradation).

This change extends `/review-council` Code Review Mode with an
optional Step 7 that transplants the posting pattern from
`/review-pr`, adapted for multi-persona aggregation.

The proposal (proposal.md) confirms PASS on all five constitution
principles. Key alignment decisions:
- **Composability**: Posting is opt-in with graceful `gh`
  degradation (Principle II)
- **Observable Quality**: Posted review includes per-persona
  provenance (Principle III)
- **Security**: JSON-to-tempfile pattern prevents shell injection
  (Principle V)

## Goals / Non-Goals

### Goals
- Add optional GitHub review posting to `/review-council`
  Code Review Mode
- Aggregate multi-persona findings into a single review body
- Transplant pre-posting checks from `/review-pr` (duplicate
  detection, stale review warnings, CODEOWNER warnings)
- Unlock review-context Protocol 2 (Issue Linking) when a PR
  is detected
- Maintain local-only behavior when no PR exists or `gh` is
  unavailable

### Non-Goals
- Deprecating `/review-pr` — the two commands serve different
  use cases (lightweight single-agent vs thorough multi-agent)
- Adding GitHub posting to Spec Review Mode — spec reviews are
  local pre-commit activities
- CI causality analysis (Phase 2 of #177) — separate concern
- Changing the fix loop behavior or iteration limits
- Adding new Divisor personas or modifying existing persona
  behavior

## Decisions

### D1: Step placement — new Step 7 after Final Report

The posting step is placed after Step 6 (Final Report), not
integrated into it. Rationale: the user sees the full local
report first, then is offered the option to share it on
GitHub. This preserves the review-then-decide flow and keeps
the Final Report step unchanged.

### D2: PR detection — `gh pr view` with argument override

PR detection uses `gh pr view --json number,headRefName,baseRefName`
to auto-detect an open PR for the current branch. The user can
also pass a PR number as `$ARGUMENTS` (e.g.,
`/review-council 42` or `/review-council code 42`). Auto-detect
runs first; explicit number overrides.

If no PR is found and no number provided, Step 7 is skipped
entirely with an informational note.

### D3: Protocol 2 conditional execution

Protocol 2 (Issue Linking) runs conditionally based on PR
number availability. There are two paths:

1. **Explicit PR number** (user passes `42` in
   `$ARGUMENTS`): The PR number is available before
   Phase 1c. Protocol 2 runs during Phase 1c alongside
   Protocols 1, 3, 4. Linked issue context enriches the
   Guard persona's prompt in Step 2.

2. **Auto-detected PR** (no explicit number): PR
   detection happens in Step 7 (after the fix loop).
   Protocol 2 cannot enrich Step 2 delegation because
   the PR number is not yet known at Phase 1c time.
   Instead, linked issue context is included only in
   the posted review body (Step 7) as additional context
   for the PR author. A future iteration may move PR
   detection earlier to enable auto-detected Protocol 2
   enrichment in Step 2.

The PR body is fetched via
`gh pr view <N> --json body --jq '.body'`. Phase 1c
currently skips Protocol 2 with the comment "This
protocol requires a PR body." This changes to: if an
explicit PR number is available from `$ARGUMENTS`, run
Protocol 2 during Phase 1c. If no PR number, skip as
before.

### D4: Multi-persona aggregation format

The review body is structured with a council summary header
followed by per-persona sections:

```
## Council Verdict: REQUEST CHANGES

**Reviewers**: Guard, Architect, Adversary, SRE, Testing
**Iterations**: 2

### Guard (REQUEST CHANGES)
- [HIGH] Finding description...

### Architect (APPROVE)
No findings.

### Adversary (REQUEST CHANGES)
- [CRITICAL] Finding description...

---
_This review was generated by /review-council (AI-assisted)._
```

Consolidated cross-persona findings (from Step 3) are presented
under the primary persona that surfaced the root cause, with
attribution to contributing personas.

### D5: Inline comment allocation — round-robin by severity

The 15-comment cap is allocated across personas using a
severity-first, round-robin strategy:

1. Collect all file-specific findings from all personas
2. Sort by severity (CRITICAL > HIGH > MEDIUM > LOW)
3. Within same severity, round-robin across personas in
   alphabetical order by persona name. When the slot
   count is odd, the first persona alphabetically
   receives the extra slot.
4. Take top 15
5. Overflow goes to the review body summary

This prevents one persona from dominating inline comment
slots. The alphabetical tie-breaking rule ensures
deterministic allocation.

### D6: Verdict mapping

| Council Verdict | GitHub Event | Notes |
|-----------------|-------------|-------|
| APPROVE | `APPROVE` | All personas unanimous |
| REQUEST CHANGES | `REQUEST_CHANGES` | Any persona dissented |
| APPROVE WITH ADVISORIES | `COMMENT` | Spec mode only; body includes advisory note |

`APPROVE WITH ADVISORIES` maps to `COMMENT` (not `APPROVE`)
because unresolved HIGH/CRITICAL advisories should not unblock
merge. The review body includes an explicit note: "Council
approved with advisories — unresolved HIGH/CRITICAL findings
require human judgment before merge."

### D7: PR number input validation

The PR number argument MUST be validated as a positive
integer (digits only, range 1–999999). Non-numeric or
out-of-range values MUST be rejected with an informational
error message before any `gh` commands execute. The
validated number is passed to `gh` commands via argument
position, not string interpolation, as defense-in-depth
per Constitution Principle V (Security by Default).

### D8: Scaffold sync

The scaffold copy at
`internal/scaffold/assets/opencode/commands/review-council.md`
MUST be updated identically to the live copy. A diff test
already exists to enforce this (drift detection).

## Risks / Trade-offs

### R1: Review body size

With 5 personas, the review body could exceed GitHub's
65,536 character limit for review bodies. Mitigation:
LOW-severity findings are summarized (count only, not
enumerated) in the review body. Only MEDIUM+ findings get
full text. If the assembled body exceeds 60,000 characters,
truncate per-persona sections starting from the persona with
the most findings, replacing detailed findings with a summary
count. Include a note: "Full findings available in the
terminal report."

### R2: APPROVE WITH ADVISORIES → COMMENT confusion

PR authors may not understand why the council "approved" but
posted as COMMENT. Mitigation: explicit note in the review body
explaining the mapping and that merge should await human review
of advisories.

### R3: Protocol 2 timing

Running Protocol 2 after the existing Phase 1c means the Guard
persona gains linked-issue context only when a PR exists. For
local-only runs (no PR), Guard still uses generic drift
detection. This is acceptable — the improvement is additive.

### R4: `gh` CLI version requirements

The `gh api` subcommand used for review posting requires `gh`
>= 2.0. Older versions may not support `--input` for JSON
payloads. Mitigation: graceful degradation with informational
error message suggesting `gh` upgrade.

### R5: Rate limiting

The posting step makes multiple `gh api` calls (review state
fetch, branch protection fetch, review posting). If `gh api`
returns HTTP 429 (rate limited), the command skips posting
gracefully with an informational message, consistent with the
existing 403/404 handling pattern.

## Testing Strategy

This change modifies only Markdown command definition files.
No new Go source code is introduced. The existing scaffold
drift detection test (`internal/scaffold/` test suite)
verifies sync between the live and scaffold copies. No
additional test coverage is required. Constitution Principle
IV (Testability) is satisfied — there is no new code surface
to cover.
