# Data Model: Specification Framework

**Spec**: 003-specification-framework
**Date**: 2026-03-08

## Entities

### 1. Scaffold Options

Configuration passed to the scaffold `Run` function.
Populated by the CLI from flags and build-time variables.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| TargetDir | string (path) | No | Target directory (default: cwd) |
| Version | string | No | Binary version (default: "dev") |
| Force | boolean | No | Overwrite all files (default: false) |

---

### 2. Scaffold Result

Outcome of a scaffold run. Tracks file disposition for
summary reporting.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| Created | string[] | Yes | Files written for the first time |
| Skipped | string[] | Yes | Files that existed and were not modified |
| Overwritten | string[] | Yes | Files replaced via `--force` |
| Updated | string[] | Yes | Tool-owned files replaced via overwrite-on-diff |

**State transitions**:
- Fresh repo: all files in `Created`
- Re-run without `--force`: user-owned in `Skipped`,
  tool-owned with changes in `Updated`
- Re-run with `--force`: all files in `Overwritten`

---

### 3. Speckit Project Configuration

Per-repository Speckit customization. Stored at
`.specify/config.yaml`.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| language | string | No | Primary language (e.g., "go", "typescript") |
| framework | string | No | Primary framework (e.g., "hugo", "react") |
| build_command | string | No | Build command (e.g., "go build ./...") |
| test_command | string | No | Test command (e.g., "go test ./...") |
| integration_patterns | string | No | Default integration style (e.g., "RESTful APIs") |
| project_type | string | No | One of: library, cli, web, mobile |
| custom_sections | object | No | Key-value overrides for template sections |

**Validation rules**:
- All fields are optional; missing fields use defaults
- `project_type` MUST be one of the allowed values if
  present
- `custom_sections` keys MUST correspond to known template
  section identifiers

**Defaults when absent**:
- `language`: language-agnostic patterns
- `framework`: none
- `build_command`: generic placeholder
- `test_command`: generic placeholder
- `integration_patterns`: "Use project-appropriate patterns"
- `project_type`: inferred from context if possible

---

### 4. OpenSpec Project Configuration

Per-repository OpenSpec customization. Stored at
`openspec/config.yaml`.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| schema | string | No | Default schema name (default: "unbound-force") |
| context | string | Yes | Project context injected into all artifact prompts; MUST include constitution reference |
| rules | object | No | Per-artifact rules (keyed by artifact ID) |

**Validation rules**:
- `context` MUST reference the org constitution
- `context` MUST NOT exceed 50KB
- `rules` keys MUST correspond to valid artifact IDs in the
  active schema (e.g., proposal, specs, design, tasks)
- `schema` MUST reference a schema that exists in
  `openspec/schemas/` or is a built-in schema

---

### 5. Custom Schema

An OpenSpec schema definition tailored to the organization.
Stored at `openspec/schemas/<name>/`.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| name | string | Yes | Schema identifier (e.g., "unbound-force") |
| artifacts | Artifact[] | Yes | Ordered list of artifact definitions |
| apply | ApplyConfig | Yes | Configuration for the apply phase |

**Artifact**:

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| id | string | Yes | Unique artifact identifier |
| generates | string | Yes | Output filename or glob pattern |
| description | string | No | Human-readable description |
| template | string | No | Template filename in templates/ |
| instruction | string | No | AI instructions for generation |
| requires | string[] | Yes | List of artifact IDs that must exist first |

**ApplyConfig**:

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| requires | string[] | Yes | Artifacts needed before apply |
| tracks | string | Yes | File used for checkbox tracking |
| instruction | string | No | AI instructions for implementation |

**Validation rules**:
- `artifacts` MUST form a valid DAG (no cycles)
- Every `requires` reference MUST point to a valid artifact
  `id`
- `template` files MUST exist in `templates/` subdirectory
- `apply.tracks` MUST reference a file generated by one of
  the artifacts

---

### 6. Pipeline Phase

One step in the Speckit pipeline.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| name | string | Yes | Phase name (e.g., "specify", "plan") |
| order | integer | Yes | Sequential position (1-9) |
| command | string | Yes | OpenCode command (e.g., "/speckit.specify") |
| prerequisites | string[] | Yes | Phase names that must complete first |
| inputs | string[] | Yes | Files that must exist before this phase |
| outputs | string[] | Yes | Files created or modified by this phase |
| is_optional | boolean | Yes | Whether the phase can be skipped |

**Validation rules**:
- `order` MUST be unique across all phases
- `prerequisites` MUST reference valid phase names
- Prerequisites MUST have lower `order` values

---

### 7. OpenSpec Action

One action in the OpenSpec tactical workflow.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| name | string | Yes | Action name (e.g., "propose", "apply") |
| command | string | Yes | OpenCode command (e.g., "/opsx:propose") |
| prerequisites | string[] | Yes | Actions or artifacts needed first |
| inputs | string[] | Yes | Files or state required |
| outputs | string[] | Yes | Files created or modified |

---

### 8. Boundary Guideline

A documented criterion for choosing between specification
tiers.

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| criterion | string | Yes | Decision criterion (e.g., "user story count") |
| speckit_when | string | Yes | When Speckit is recommended |
| openspec_when | string | Yes | When OpenSpec is recommended |
| examples | string[] | No | Illustrative examples |

## Relationships

```text
Scaffold Options

Scaffold Result
  +-- tracks --> Scaffolded files (1:N)

Speckit Project Configuration
  +-- consumed by --> Pipeline Phase (1:N)
  +-- resides in --> Consumer Repository (1:1)

OpenSpec Project Configuration
  +-- references --> Custom Schema (1:1)
  +-- references --> Constitution (1:1)
  +-- consumed by --> OpenSpec Action (1:N)
  +-- resides in --> Consumer Repository (1:1)

Custom Schema
  +-- contains --> Artifact[] (1:N)
  +-- contains --> ApplyConfig (1:1)
  +-- embedded in --> Speckit binary (1:1)

Pipeline Phase
  +-- depends on --> Pipeline Phase (N:N via prerequisites)

Boundary Guideline
  +-- recommends --> Pipeline Phase | OpenSpec Action (1:1)
```
