# Feature Specification: Swarm Delegation Workflow

**Feature Branch**: `012-swarm-delegation`
**Created**: 2026-03-22
**Status**: Draft
**Input**: User description: "Delegate to the swarm (all
unbound heroes) after /speckit.clarify through
/review-council, so the swarm returns to let a human
accept. Include a retrospective/learning step with
empirical data from all agents."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Swarm Delegation After Clarify (Priority: P1)

A human operator specifies and clarifies a feature using
the Speckit pipeline (`/speckit.specify` and
`/speckit.clarify`). Once the specification is clear, the
operator hands off to the swarm. The swarm autonomously
runs planning, implementation, validation, and review --
all hero stages between clarify and acceptance. The
workflow pauses automatically before the acceptance stage,
returning control to the human for a decision.

**Why this priority**: This is the core value proposition.
Without execution mode awareness and automatic checkpoint
pausing, the swarm cannot operate autonomously and return
control at the right moment. Every other story depends on
this handoff mechanism.

**Independent Test**: Can be fully tested by starting a
workflow, advancing through the define stage manually,
then observing that the swarm-mode stages advance without
human intervention and the workflow pauses before the
accept stage. Delivers the autonomous delegation loop.

**Acceptance Scenarios**:

1. **Given** a workflow where the define stage is
   completed (spec is clarified), **When** the workflow
   advances past define, **Then** the implement, validate,
   and review stages are marked as swarm-execution-mode
   and the workflow proceeds through them autonomously.
2. **Given** a workflow advancing from the review stage
   (swarm mode) to the accept stage (human mode), **When**
   the swarm completes review, **Then** the workflow
   status transitions to "awaiting_human" and the accept
   stage remains pending until the human explicitly
   advances.
3. **Given** a workflow in "awaiting_human" status,
   **When** the human runs `/workflow advance`, **Then**
   the accept stage is activated and the workflow status
   returns to "active".

---

### User Story 2 - Execution Mode Per Stage (Priority: P1)

Each workflow stage carries an execution mode attribute
indicating whether it is driven by a human operator or by
the swarm autonomously. The default mode assignments are:
define (human), implement (swarm), validate (swarm),
review (swarm), accept (human), reflect (swarm). The
execution mode is visible in workflow status output so
operators understand which stages require their attention
and which run automatically.

**Why this priority**: The execution mode is the data
model foundation for the delegation feature. Without it,
the engine cannot distinguish swarm-driven stages from
human-driven stages, and the checkpoint logic has no
signal to pause on.

**Independent Test**: Can be fully tested by creating a
new workflow and inspecting the persisted JSON to confirm
each stage has the correct execution mode. Status output
can be checked for mode indicators.

**Acceptance Scenarios**:

1. **Given** a newly started workflow with all heroes
   available, **When** the workflow JSON is inspected,
   **Then** each stage has an `execution_mode` field set
   to `"human"` or `"swarm"` per the default assignments.
2. **Given** a legacy workflow JSON file without
   `execution_mode` fields, **When** the workflow is
   loaded by the engine, **Then** empty execution mode is
   treated as `"human"` for backward compatibility and the
   workflow advances stage-by-stage as before.
3. **Given** a workflow in progress, **When** the user
   runs `/workflow status`, **Then** each stage displays
   its execution mode alongside the hero name and status.

### User Story 3 - Reflect Stage (Priority: P2)

The final workflow stage is renamed from "measure" to
"reflect." This stage is owned by Mx F and runs
autonomously after the human accepts the increment. It
produces a richer output than the previous measure stage:
a metrics snapshot (velocity, quality trends, CI health),
cross-hero learning feedback generated by analyzing
workflow records for recurring patterns, and a
retrospective summary that incorporates empirical data
from Gaze's quality report and the Divisor's review
verdict. The reflect stage explicitly consumes artifacts
produced by the validate and review stages.

**Why this priority**: The reflect stage enriches the
workflow's learning loop but is not required for the core
delegation mechanism. It can be implemented after the
execution mode and checkpoint features are working.

**Independent Test**: Can be fully tested by completing a
full workflow and verifying that the final stage is named
"reflect," produces a metrics snapshot and learning
feedback artifacts, and references quality report and
review verdict data from earlier stages.

**Acceptance Scenarios**:

1. **Given** a workflow where the accept stage is
   completed, **When** the workflow advances to the final
   stage, **Then** the stage is named "reflect" (not
   "measure") and is assigned to Mx F.
2. **Given** a workflow completing the reflect stage,
   **When** the stage produces its output, **Then** it
   includes a metrics snapshot and triggers the learning
   analysis engine (`AnalyzeWorkflows`) to detect
   cross-hero patterns.
3. **Given** the reflect stage in a completed workflow,
   **When** the workflow record is generated, **Then** the
   record references artifacts consumed from the validate
   stage (quality report) and review stage (review
   verdict).
4. **Given** at least 3 completed workflow records,
   **When** the reflect stage runs learning analysis,
   **Then** it produces `LearningFeedback` artifacts with
   actionable recommendations (e.g., convention pack
   updates for recurring review findings).

---

### User Story 4 - Workflow Status Shows Execution Mode (Priority: P3)

The `/workflow status` command displays execution mode
information for each stage, making it clear which stages
are human-driven and which are swarm-driven. The
`/workflow start` command also shows this in its initial
output. The awaiting_human status has a distinct visual
indicator in status output.

**Why this priority**: This is a usability enhancement
that helps operators understand the workflow shape at a
glance. The core functionality works without it, but the
operator experience is significantly better with clear
mode indicators.

**Independent Test**: Can be fully tested by starting a
workflow and checking the terminal output for execution
mode labels and status indicators.

**Acceptance Scenarios**:

1. **Given** a started workflow, **When** the user runs
   `/workflow status`, **Then** each stage row includes a
   mode indicator (e.g., `[human]` or `[swarm]`).
2. **Given** a workflow in "awaiting_human" status,
   **When** the user runs `/workflow status`, **Then** the
   status shows "awaiting_human" and the next pending
   stage (accept) is highlighted as requiring human
   action.
3. **Given** a started workflow, **When** the user runs
   `/workflow start`, **Then** the initial output includes
   execution mode per stage in the hero availability list.

### Edge Cases

- What happens when a swarm-mode stage's hero is
  unavailable? The stage is skipped (existing behavior).
  If all swarm-mode stages between two human checkpoints
  are skipped, the workflow transitions directly to the
  next human-mode stage.
- What happens when the human advances a workflow that is
  already in "awaiting_human" status? The accept stage is
  activated and the workflow returns to "active" status.
  This is the normal resume path.
- What happens when a legacy workflow JSON (without
  `execution_mode` fields) is loaded? All stages are
  treated as human-mode for backward compatibility. The
  workflow advances one stage at a time as before, with no
  automatic checkpoint pausing.
- How does the review-implement loop interact with
  execution modes? The existing escalation logic (max 3
  iterations) is unchanged. If the review stage triggers
  escalation, the workflow is escalated regardless of
  execution mode.
- What happens if the define stage is the only
  human-mode stage at the start? After the human advances
  past define, the swarm runs through all swarm-mode
  stages (implement, validate, review) and pauses at
  accept (the next human-mode stage).
- What happens when the reflect stage (swarm mode) is the
  last stage after accept? After the human accepts, the
  swarm runs reflect autonomously and the workflow
  completes without further human intervention.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Each workflow stage MUST carry an
  `execution_mode` attribute with value `"human"` or
  `"swarm"`.
- **FR-002**: The default execution mode assignments MUST
  be: define=human, implement=swarm, validate=swarm,
  review=swarm, accept=human, reflect=swarm.
- **FR-003**: The orchestration engine MUST introduce a
  new workflow status `"awaiting_human"` that represents
  a normal, expected pause point (distinct from
  escalation or failure).
- **FR-004**: When the `Advance()` method completes a
  swarm-mode stage and the next non-skipped stage is
  human-mode, the workflow MUST transition to
  `"awaiting_human"` status instead of activating the
  next stage.
- **FR-005**: When the `Advance()` method is called on a
  workflow in `"awaiting_human"` status, it MUST activate
  the pending human-mode stage and set the workflow status
  back to `"active"`.
- **FR-006**: The `"measure"` stage MUST be renamed to
  `"reflect"` across the stage constants, stage ordering,
  hero mapping, and all documentation.
- **FR-007**: The reflect stage MUST be assigned to Mx F
  and MUST explicitly consume artifacts from the validate
  stage (quality report) and review stage (review
  verdict).
- **FR-008**: The reflect stage MUST trigger the learning
  analysis engine (`AnalyzeWorkflows`) to detect
  cross-hero patterns when sufficient workflow records
  exist (3 or more).
- **FR-009**: The reflect stage MUST produce both a
  metrics snapshot and learning feedback artifacts
  alongside the existing workflow-record artifact.
  (Verified through SKILL.md documentation review and
  end-to-end workflow testing. The Go engine tracks
  `ArtifactsProduced` metadata but does not enforce
  artifact production -- see Assumptions.)
- **FR-010**: Legacy workflow JSON files without
  `execution_mode` fields MUST be handled gracefully.
  Empty or missing execution mode MUST be treated as
  `"human"` for backward compatibility.
- **FR-011**: The `/workflow status` command MUST display
  execution mode per stage and MUST show a distinct
  indicator for `"awaiting_human"` status.
- **FR-012**: The `/workflow start` command MUST include
  execution mode information in its output.
- **FR-013**: The `/workflow advance` command MUST
  document the checkpoint behavior and the
  `"awaiting_human"` output format.
- **FR-014**: When all swarm-mode stages between two
  human checkpoints are skipped (heroes unavailable), the
  workflow MUST transition directly to the next
  human-mode stage without entering `"awaiting_human"`.

### Key Entities

- **Execution Mode**: An attribute on each workflow stage
  indicating whether the stage is driven by a human
  operator (`"human"`) or by the swarm autonomously
  (`"swarm"`). Determines checkpoint behavior at stage
  transitions.
- **Human Checkpoint**: A transition point where the
  workflow pauses (status `"awaiting_human"`) because the
  next stage requires human involvement. Occurs at
  swarm-to-human mode boundaries.
- **Reflect Stage**: The renamed final stage (was
  "measure") that produces metrics, learning feedback,
  and a retrospective summary incorporating empirical
  data from all hero artifacts.

## Assumptions

- The Swarm coordinator (external tool) is responsible
  for actually executing the hero logic within each
  swarm-mode stage. The orchestration engine only tracks
  state and manages transitions -- it does not invoke
  heroes directly.
- The reflect stage's richer behavior (consuming
  artifacts from validate and review, running learning
  analysis) is described in the SKILL.md for the Swarm
  coordinator to follow. The Go engine tracks which
  artifacts are consumed/produced via metadata, not by
  executing the analysis itself.
- The learning analysis threshold (3 completed workflows
  minimum) remains unchanged from the existing
  `AnalyzeWorkflows` implementation.
- The reflect stage remains owned by Mx F. No new hero
  is introduced.
- The 6-stage count is preserved. This is a rename and
  enrichment, not a stage addition.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A workflow started with all heroes available
  pauses exactly once (at the swarm-to-human boundary
  before accept) without human intervention during the
  swarm-mode stages.
- **SC-002**: Legacy workflows without execution mode
  fields continue to function identically to pre-change
  behavior -- 100% backward compatibility for existing
  persisted JSON files.
- **SC-003**: The reflect stage produces at least 2
  artifact types (metrics snapshot and learning feedback
  or workflow record) on every completed workflow.
- **SC-004**: All existing orchestration tests pass after
  the rename and execution mode additions, with zero
  regressions.
- **SC-005**: The `/workflow status` output for a paused
  workflow clearly indicates which stage requires human
  action, distinguishable from error or escalation states.
- **SC-006**: The complete end-to-end workflow (define
  through reflect) completes with no more than 2 human
  decision points: one to hand off after define, and one
  to accept the increment. The workflow enters
  `"awaiting_human"` status exactly once (before the
  accept stage). Note: the human issues 3 `/workflow
  advance` calls total (define handoff, accept resume,
  accept completion), but only 2 represent decision
  points.
