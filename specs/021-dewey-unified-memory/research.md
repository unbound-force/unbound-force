# Research: Dewey Unified Memory

**Branch**: `021-dewey-unified-memory`
**Date**: 2026-04-03

## R1: Ollama Subprocess Management

**Question**: How should Dewey manage the Ollama
subprocess lifecycle?

**Decision**: Detect-then-start with fire-and-forget
exit semantics.

**Rationale**: On startup, `dewey serve` probes the
Ollama HTTP health endpoint (`GET /` on
`localhost:11434`). Three outcomes:

1. **Ollama responding**: Use existing instance. Set
   `ollamaManaged = false`. No subprocess started.
2. **Ollama not responding, binary in PATH**: Start
   `ollama serve` as a detached subprocess. Poll the
   health endpoint (max 30s, 500ms intervals). Set
   `ollamaManaged = true`. On timeout, fall back to
   keyword-only mode.
3. **Ollama not responding, binary not in PATH**: Log
   informational message. Set keyword-only mode. No
   subprocess started.

On `dewey serve` exit: leave Ollama running regardless
of `ollamaManaged` flag. Other consumers (other Dewey
instances, direct `ollama` CLI usage, Swarm Hivemind)
may depend on it. Killing Ollama on exit would violate
the Composability First principle — Dewey should not
assume exclusive ownership of shared infrastructure.

**Race condition mitigation**: Two concurrent
`dewey serve` instances may both detect Ollama as not
running and both attempt to start it. The second
`ollama serve` will fail to bind the port (EADDRINUSE)
and exit. The second Dewey instance's health poll will
then succeed (the first instance's Ollama is now
serving). No coordination protocol needed — the OS
port binding provides natural mutual exclusion.

**Alternatives rejected**:
- PID file locking: Over-engineered for this use case.
  Port binding already provides mutual exclusion.
- Kill Ollama on exit: Violates Composability First.
  Other tools may depend on Ollama.
- Always start a new Ollama: Wasteful when one is
  already running. Violates the "detect existing
  instance" requirement (FR-005).

**Implementation location**: Dewey repo
(`unbound-force/dewey`), likely in a new
`internal/ollama/` package.

## R2: Learning Document Schema

**Question**: How should learnings be stored in Dewey's
index alongside specs, code, and web documentation?

**Decision**: Learnings are documents with
`source_type: "learning"` stored in Dewey's existing
document index. They use the same embedding pipeline as
all other documents.

**Schema**:

```
Learning {
  id:          string     // Auto-generated UUID
  content:     string     // The learning text
  tags:        []string   // User-provided tags (branch, date, category)
  source_type: "learning" // Distinguishes from "disk", "github", "web"
  source_id:   string     // "learning-{uuid}" for provenance
  created_at:  timestamp  // When stored
  embedding:   []float32  // Generated by embedding model
}
```

**Rationale**: Dewey already has a document abstraction
with `source_type` metadata. Learnings are simply
another source type. This means:
- `dewey_semantic_search` returns learnings alongside
  specs and code — unified search with zero extra work.
- `dewey_semantic_search_filtered` with
  `source_type: "learning"` returns only learnings.
- No new query tools needed. Existing tools work.

**MCP tool**: `dewey_store_learning` — a new MCP tool
that accepts `{ text: string, tags: string[] }` and
returns `{ id: string, indexed: boolean }`.

**Persistence**: Learnings are stored in `graph.db`
(SQLite). They survive across sessions. They are NOT
backed up to JSONL (unlike Hivemind's
`memories.jsonl`). If `graph.db` is corrupted,
learnings are lost. Future consideration: periodic
JSONL export for backup.

**Alternatives rejected**:
- Separate learning database: Violates the "unified"
  principle. Would require a separate query tool.
- JSONL file as primary store: Not indexed. Would
  require re-embedding on every startup.
- Markdown files in `.dewey/learnings/`: Would work
  for disk source indexing but adds filesystem
  management complexity. Learnings are ephemeral
  enough that database storage is appropriate.

## R3: Hivemind Deprecation Strategy

**Question**: How should the transition from Hivemind
to Dewey be managed?

**Decision**: Hard migration in this repo's Markdown
files. Soft deprecation in the Swarm fork.

**This repo (agent migration)**:
- Replace all `hivemind_store` references with
  `dewey_store_learning` in `/unleash` command.
- Replace all `hivemind_find` references with
  `dewey_semantic_search` in Divisor agents.
- Update graceful degradation: "If Dewey is available"
  replaces "If Hivemind is available."
- No backward compatibility needed — these are
  instruction files, not APIs.

**Swarm fork (Hivemind tools)**:
Two viable approaches, decided at fork time:

1. **Proxy approach**: `hivemind_store` internally
   calls `dewey_store_learning`. `hivemind_find`
   internally calls `dewey_semantic_search`. Existing
   Swarm workflows continue working transparently.
   Pros: Zero breaking changes. Cons: Maintains two
   tool names for the same operation.

2. **Deprecation approach**: `hivemind_store` and
   `hivemind_find` return error messages with migration
   instructions: "Use dewey_store_learning instead."
   Pros: Clean break. Cons: Breaks existing workflows
   until agent files are updated.

**Recommendation**: Start with proxy approach (option
1) for the initial fork. Migrate agent files. Then
switch to deprecation (option 2) once all consumers
are updated. This is a two-phase migration within the
fork.

**No data migration**: Existing Hivemind learnings in
`.hive/memories.jsonl` and SQLite are abandoned. New
learnings go to Dewey from day one. The volume of
existing learnings is small enough that re-learning
is acceptable.

## R4: Swarm Fork Mechanics

**Question**: How should the Swarm plugin fork be
structured?

**Decision**: GitHub fork of the upstream Swarm plugin
repository into `unbound-force/swarm`. Install via
npm/bun from the GitHub repository URL.

**Fork process**:
1. Fork upstream repo to `unbound-force/swarm` via
   GitHub UI or `gh repo fork`.
2. Update `package.json` name to
   `@unbound-force/opencode-swarm-plugin` (scoped).
3. Modify Hivemind tools to proxy through Dewey (per
   R3 decision).
4. Publish to npm under the `@unbound-force` scope, or
   install directly from GitHub.

**Installation change**:
- Current: `npm install -g opencode-swarm-plugin@latest`
- After fork: `npm install -g @unbound-force/opencode-swarm-plugin@latest`
  or `npm install -g github:unbound-force/swarm`

**Files affected in this repo**:
- `internal/setup/setup.go`: Update install command
  from `opencode-swarm-plugin@latest` to the forked
  package name.
- `internal/doctor/environ.go`: Update install hints
  for `swarm` tool.
- `opencode.json`: Update plugin array entry if the
  package name changes.

**Upstream relationship**: The fork is the project's
responsibility. Upstream changes can be cherry-picked
as needed. No automatic merge. The fork diverges
intentionally for Hivemind modifications.

**Alternatives rejected**:
- Monkey-patching upstream: Not possible with npm
  global installs. No extension points for Hivemind.
- Wrapper plugin: Would need to intercept MCP tool
  calls. More complex than a fork.
- Contributing upstream: The Hivemind changes are
  specific to Unbound Force's architecture. Upstream
  is unlikely to accept Dewey-specific modifications.

## R5: Doctor Health Check Consolidation

**Question**: How should `uf doctor` health checks
change when Dewey manages Ollama?

**Decision**: Consolidate Ollama checks under the
Dewey Knowledge Layer group. Add embedding capability
check. Demote standalone Ollama check.

**Current state** (3 places checking Ollama):
1. Core Tools group: `ollama` binary check + model
   check (`checkOllamaModel`).
2. Dewey Knowledge Layer group: `dewey binary` +
   `embedding model` (via `ollama list`) + `workspace`.
3. Swarm Plugin group: indirect (Swarm uses Ollama
   via Hivemind).

**After this spec**:
1. Core Tools group: `ollama` binary check remains
   (informational, not required). Model check remains
   as-is (still useful for standalone Ollama users).
2. Dewey Knowledge Layer group: Add `embedding
   capability` check that verifies Dewey can actually
   generate embeddings (not just that Ollama is
   installed). This is a functional check, not just a
   binary existence check.
3. Swarm Plugin group: No changes (Swarm fork handles
   its own Ollama dependency via Dewey proxy).

**New check**: `checkDeweyEmbeddings` — when Dewey
binary is found, attempt to verify embedding
capability. Options:
- Run `dewey status` and parse output for embedding
  readiness.
- Check if Ollama is serving AND the model is pulled
  (current approach, but now framed as "Dewey can
  generate embeddings" rather than "Ollama model
  available").

**Decision**: Keep the existing check structure but
reframe the messaging. The Dewey group's "embedding
model" check already does the right thing (checks
`ollama list` for `granite-embedding`). Add a note
in the output: "Dewey manages Ollama lifecycle —
embeddings available" vs "Dewey running but embeddings
unavailable (keyword-only mode)."

FR-021 says the direct Ollama check MAY be removed.
Decision: Keep it in Core Tools for now (it's
informational and helps debug Ollama issues
independently of Dewey). Revisit in a future spec if
it causes confusion.
