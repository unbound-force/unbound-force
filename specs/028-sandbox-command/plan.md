# Implementation Plan: Sandbox Command

**Branch**: `028-sandbox-command` | **Date**: 2026-04-12 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/028-sandbox-command/spec.md`

## Summary

Add a `uf sandbox` CLI command with 5 subcommands (`start`,
`stop`, `attach`, `extract`, `status`) that orchestrate
Podman containers for isolated OpenCode sessions. The
implementation creates a new `internal/sandbox/` package
following the established `Options`/`ExecCmd` injection
pattern and registers a new Cobra command group in
`cmd/unbound-force/`. Platform detection handles macOS
arm64 vs Fedora amd64 with automatic SELinux `:Z` flag
application. Two mount modes are supported: isolated
(read-only with overlay, default) and direct (read-write).
Change extraction uses `git format-patch` / `git am` for
round-trip patch workflows.

## Technical Context

**Language/Version**: Go 1.24+
**Primary Dependencies**: `github.com/spf13/cobra` (CLI), `github.com/charmbracelet/log` (logging), `net/http` (health check), `os/exec` (subprocess)
**Storage**: N/A (no persistent state — container lifecycle is transient)
**Testing**: Standard library `testing` package with injected dependencies
**Target Platform**: macOS arm64, Fedora amd64 (Linux)
**Project Type**: CLI (extension to existing `uf` binary)
**Performance Goals**: Start-to-attach in < 30 seconds (excluding image pull) per SC-001
**Constraints**: Single container at a time (FR-016), 60-second health check timeout (FR-005)
**Scale/Scope**: 5 subcommands, ~3 new Go source files, ~1 new Cobra command file

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### I. Autonomous Collaboration — PASS

The sandbox command is a standalone CLI capability. It does
not require any other hero to function. It does not produce
inter-hero artifacts — it is a developer tool that
orchestrates containers. No artifact envelope, no hero
coupling.

### II. Composability First — PASS

The sandbox command is independently usable. It requires
only Podman and OpenCode — both are external tools, not
heroes. The command does not introduce mandatory
dependencies on other heroes. When Ollama is unavailable,
the sandbox still starts (with a warning). The command
auto-detects platform capabilities without requiring
manual configuration.

### III. Observable Quality — PASS

The `status` subcommand provides machine-inspectable
container state. Error messages include actionable hints
(install commands, manual steps). The `extract` subcommand
shows a diffstat for human review before applying changes.
All output goes through `io.Writer` for testability and
potential JSON formatting in the future.

### IV. Testability — PASS

All external dependencies (`LookPath`, `ExecCmd`,
`ExecInteractive`, `Getenv`, `HTTPGet`) are injectable
on the `Options` struct following the established pattern
from `internal/setup/` and `internal/doctor/`. Tests use
injected stubs — no real Podman, no real containers, no
network access, no shared mutable state. Platform
detection uses injectable `ExecCmd` for `getenforce`.

**Coverage strategy**: Unit tests for all public functions
and key internal functions. Target ≥ 80% line coverage for
`internal/sandbox/`. Pure logic functions (`detect.go`,
`config.go`) target ≥ 90%. See
[contracts/testing-strategy.md](contracts/testing-strategy.md)
for the full test plan.

## Design Decisions

### D1: Single sandbox.go Cobra file vs. per-subcommand files

**Decision**: Single `cmd/unbound-force/sandbox.go` file
containing `newSandboxCmd()` and all 5 subcommand
constructors.

**Rationale**: The existing codebase puts each command in
`main.go` directly. A separate file for the sandbox
command group keeps `main.go` clean while grouping related
subcommands. Per-subcommand files would create 5 tiny
files — unnecessary fragmentation for a cohesive command
group.

### D2: ExecInteractive for opencode attach

**Decision**: Add an `ExecInteractive` function field to
`Options` that connects stdin/stdout/stderr to the
terminal, separate from `ExecCmd` which captures output.

**Rationale**: `opencode attach` is an interactive TUI
command that needs direct terminal access. The existing
`ExecCmd` pattern captures output as `[]byte`, which is
wrong for interactive commands. A separate injection point
keeps the two patterns clean and independently testable.

### D3: Health check via HTTP vs. podman exec

**Decision**: Use direct HTTP GET from the host to
`http://localhost:4096` for health checking.

**Rationale**: The port is already mapped (`-p 4096:4096`),
so direct HTTP is simpler than `podman exec curl ...`.
The `HTTPGet` function is injectable for testing. This
avoids requiring `curl` inside the container.

### D4: Patch extraction via git format-patch

**Decision**: Use `git format-patch` (preserves commits)
rather than `git diff` (loses commit metadata).

**Rationale**: Engineers want to see individual commits
from the agent, not a single squashed diff. `git am`
applies patches as proper commits with author and message
preserved. Fall back to `git diff` only if `format-patch`
fails (e.g., no commits, only staged changes).

### D5: No persistent state

**Decision**: The sandbox command stores no persistent
state. Container existence is the state.

**Rationale**: `podman inspect` tells us everything we
need to know. Adding a state file (like workflow JSON)
would create synchronization issues — what if the
container is stopped externally? Querying Podman directly
is always authoritative. This follows the principle of
a single source of truth.
