# Implementation Plan: Sandbox CDE Lifecycle

**Branch**: `029-sandbox-cde-lifecycle` | **Date**: 2026-04-13 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/029-sandbox-cde-lifecycle/spec.md`

## Summary

Extend the existing `internal/sandbox/` package (Spec 028)
with a persistent workspace lifecycle (`create`/`destroy`)
and a CDE backend (Eclipse Che / Dev Spaces). The
implementation introduces a `Backend` interface with two
implementations: `PodmanBackend` (named volumes for
persistent state) and `CheBackend` (workspace provisioning
via `chectl` CLI or Che REST API). Bidirectional git sync
replaces the extract workflow for persistent workspaces.
Demo endpoints are exposed via devfile endpoints (CDE) or
port mappings (Podman). All existing Spec 028 behavior is
preserved as the ephemeral fallback mode.

## Technical Context

**Language/Version**: Go 1.24+
**Primary Dependencies**: `github.com/spf13/cobra` (CLI),
`github.com/charmbracelet/log` (logging), `net/http`

`gopkg.in/yaml.v3` (sandbox config parsing)
**Storage**: N/A (Podman named volumes and CDE platform
manage persistence; no application-level state files)
**Testing**: Standard library `testing` package with
injected dependencies
**Target Platform**: macOS arm64, Fedora amd64 (Linux)
**Project Type**: CLI (extension to existing `uf` binary)
**Performance Goals**: Create-to-ready in < 60 seconds
(excluding image pull / CDE provisioning)
**Constraints**: Backward compatible with Spec 028
ephemeral mode (FR-018). Single workspace per project
(relaxed from Spec 028's single global container).
**Scale/Scope**: 2 new subcommands, ~4 new Go source
files, ~1 updated Cobra command file, ~1 config file
format

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after
Phase 1 design.*

### I. Autonomous Collaboration — PASS

The sandbox command is a standalone CLI capability. It
does not produce inter-hero artifacts. The CDE backend
communicates with Eclipse Che via `chectl` or REST API —
these are external infrastructure tools, not heroes.
No artifact envelope, no hero coupling.

### II. Composability First — PASS

The sandbox command is independently usable. CDE backend
is optional — when Che is not configured, the Podman
backend is used automatically (auto-detection per
FR-004). The `Backend` interface enables adding new
backends without modifying existing code (Open/Closed
Principle). Each backend is independently testable.
When Ollama is unavailable, the sandbox still starts
(with a warning). When `chectl` is unavailable but
`UF_CHE_URL` is set, the REST API fallback activates.

### III. Observable Quality — PASS

The `status` subcommand provides machine-inspectable
workspace state for both backends. Error messages include
actionable hints (install commands, configuration steps).
Demo endpoint URLs are displayed for review. All output
goes through `io.Writer` for testability and potential
JSON formatting in the future.

### IV. Testability — PASS

All external dependencies are injectable on the `Options`
struct following the established pattern from Spec 028.
The `Backend` interface itself is injectable — tests can
provide a mock backend. `PodmanBackend` and `CheBackend`
use the same `ExecCmd`/`LookPath`/`HTTPGet` injection
points. Config file parsing uses injectable `ReadFile`.
No real Podman, Che, containers, or network access in
tests.

**Coverage strategy**: Unit tests for all public
functions and key internal functions. Target ≥ 80% line
coverage for `internal/sandbox/`. Pure logic functions
(`backend.go`, `workspace.go`) target ≥ 90%.
Orchestration functions (`podman.go`, `che.go`) target
≥ 75%. See
[contracts/testing-strategy.md](contracts/testing-strategy.md)
for the full test plan with 37+ named test cases.

## Design Decisions

### D1: Backend Interface vs. Switch Statement

**Decision**: Use a `Backend` interface with two
implementations (`PodmanBackend`, `CheBackend`) rather
than a switch statement on backend name.

**Rationale**: The Strategy pattern (SOLID Open/Closed
Principle) allows adding new backends without modifying
the orchestration layer. The interface is also injectable
for testing — mock backends can verify orchestration
logic without real infrastructure. See research R1.

### D2: Named Volumes vs. Bind Mounts for Persistence

**Decision**: Use Podman named volumes
(`podman volume create uf-sandbox-<project>`) for
persistent state instead of bind mounts to a host
directory.

**Rationale**: Named volumes are managed by Podman and
avoid permission issues on macOS (Podman VM boundary)
and SELinux complications on Fedora. They survive
container removal and can be inspected/cleaned up via
standard Podman commands. See research R3.

### D3: Per-Project Container Names

**Decision**: Change the container name from the fixed
`uf-sandbox` (Spec 028) to `uf-sandbox-<project-name>`
for persistent workspaces. Ephemeral mode retains the
fixed name for backward compatibility.

**Rationale**: Supports multiple concurrent sandboxes
(one per project) per the spec edge case. The project
name is derived from the directory basename, sanitized
to lowercase alphanumeric + hyphens. See research R4.

### D4: chectl Primary, REST API Fallback

**Decision**: Support both `chectl` CLI and Che REST API
for CDE workspace management. `chectl` is the primary
path; REST API is the fallback when `chectl` is not
installed but `UF_CHE_URL` is set.

**Rationale**: `chectl` handles authentication, endpoint
discovery, and error formatting. The REST API fallback
supports environments where only the server URL is
available (CI/CD, remote servers). See research R2.

### D5: Git Sync as Coordination Mechanism

**Decision**: Use standard git push/pull for
bidirectional sync between host and workspace. No custom
sync protocol.

**Rationale**: Both the agent and engineer already use
git. The workspace has the same remote origin. Conflict
detection uses `git pull --ff-only`. This is simpler and
more reliable than `rsync`, `podman cp`, or custom file
watchers. See research R5.

### D6: Configuration File (.uf/sandbox.yaml)

**Decision**: Add a persistent configuration file at
`.uf/sandbox.yaml` for CDE URL, Ollama endpoint, default
backend, and demo port mappings.

**Rationale**: CDE configuration (URL, token) should
persist across sessions and be version-controllable.
Environment variables provide overrides for CI/CD.
The precedence order is: CLI flag > env var > config
file > default. See research R7.

### D7: Backward Compatibility Strategy

**Decision**: The existing `uf sandbox start` (ephemeral
mode) continues to work exactly as before. Persistent
mode is activated only when a named volume exists (from
a prior `create`). Detection is automatic.

**Rationale**: FR-018 requires all existing tests and
subcommands to continue working. Engineers who don't
use `create`/`destroy` see no behavior change. The
detection logic (check for named volume) is simple and
deterministic. See research R9.

## Post-Design Constitution Re-evaluation

### I. Autonomous Collaboration — PASS (confirmed)

No inter-hero artifacts introduced. The Backend
interface is internal to the sandbox package. CDE
communication uses standard external tools (chectl,
REST API), not hero-to-hero protocols.

### II. Composability First — PASS (confirmed)

The CDE backend is purely additive. The Podman backend
works without Che. The persistent lifecycle works without
CDE. Each capability layer is independently useful:
- Podman ephemeral (Spec 028) — works alone
- Podman persistent (named volumes) — works without CDE
- CDE (Eclipse Che) — works without Podman

Auto-detection ensures the best available backend is
selected without manual configuration.

### III. Observable Quality — PASS (confirmed)

`WorkspaceStatus` provides structured state for both
backends. Demo endpoint URLs are discoverable via
`status`. Error messages are actionable with install
hints and configuration suggestions. All output goes
through `io.Writer`.

### IV. Testability — PASS (confirmed)

The `Backend` interface is the primary testability
enabler — mock backends can verify orchestration logic
without infrastructure. Each backend implementation
uses the same injectable `Options` pattern from Spec
028. The testing strategy document specifies 37+ test
cases covering all public functions, error paths,
backend selection, and backward compatibility. Coverage
targets are defined (≥ 80% overall, ≥ 90% for pure
logic). No integration tests requiring real containers
or CDE instances.
